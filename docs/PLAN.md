# PLAN.md — にゃんこタイピング

## 1. プロジェクトの目的

### なぜこのゲームを作るのか

- **タイピングの「入り口」を用意する**  
  小学校での ICT 活用や家庭での PC 利用が増える中、**ローマ字とキーボードに初めて触れる子**が「楽しい」と感じられる体験を届けたい。  
  にゃんこタイピングは、**ゲームとしての達成感**と**タイピングスキルの土台**を同時に提供することを目的としています。

- **「練習」ではなく「遊び」として届ける**  
  単なるドリルではなく、**敵を倒す・クリアする**というゲーム体験にのせて、自然に指の動きとローマ字の対応を覚えてもらう。  
  「にゃんこ大戦争」ライクな見た目とルールで、子どもが自分から「もう1回やりたい」と思えるプロトタイプを目指します。

- **教育とエンタメの接点**  
  教育用ゲームとして、**正しいキー入力の習慣**と**ローマ字（ふりがな⇔アルファベット）の対応**を、負荷の少ない「2〜6文字タイピング」から積み上げられる設計にします。

## 2. ターゲットユーザー層

### 主なターゲット

| 層 | 想定 | 配慮のポイント |
|----|------|----------------|
| **小学生（低〜中学年）** | タイピング・ローマ字を習い始めたばかり | 表示は大きく、ふりがな＋大文字ローマ字で「読める・打ちやすい」ようにする |
| **タイピング入門者** | キーボードの位置がまだ曖昧 | 1文字ずつの判定で負荷を下げ、ミスしてもやり直しやすいフィードバック |
| **にゃんこ大戦争を知っている子** | タワーデフェンスの雰囲気に親しみ | 自陣・敵陣の構図と「敵を倒す」感覚で、ルールを直感的に理解しやすくする |

### 入門者への配慮（方針）

- **視認性**: 現在の入力対象（ふりがな＋大文字ローマ字）を表示する。プロトタイプでは**上部UIの枠内にのみ**表示し、敵の頭上には表示しない。小さい文字や長い単語は避ける。
- **操作**: キーボードのみで完結。マウスは補助（スタート・リトライなど）に留める。
- **難易度**: プロトタイプでは**2文字タイピング**に限定し、「どのキーを押すか」に集中できるようにする。
- **フィードバック**: 入力した文字の色を白→グレーに変え、正解・撃破が分かりやすい演出にする。

## 3. コア体験（UX）

### プレイ中にユーザーにどう感じてほしいか

1. **「わかった！」**  
   画面上のローマ字を見て、対応するキーを押すと敵に当たる／文字が変わる。**因果がはっきりしている**ことで、ルールをすぐ理解できる。

2. **「できた！」**  
   1体ずつ敵を撃破し、全10体クリアしたときの達成感。**短いセッションで完結**し、「もう1回」につなげやすい。

3. **「にゃんこっぽい！」**  
   自陣（右）・敵陣（左）の構図、敵が右へ進む動き、シンプルでポップな見た目で、**知っているゲームの空気感**を感じられる。

4. **「間違えても大丈夫」**  
   1文字単位の判定と色の変化で、どこまで打ったかが一目で分かる。ミスしても**やり直しがしやすく**、挫折しにくい体験にする。

## 4. プロトタイプのゴール

### 「プロトタイプ完了」の定義

以下がすべて満たされた状態を、**プロトタイプ版の完了**とします。

| 項目 | 完了条件 |
|------|----------|
| **スタート画面** | タイトル・開始ボタン・メニューが表示され、開始でゲームプレイ画面へ遷移できる |
| **ゲームプレイ画面** | にゃんこ大戦争風のレイアウト（自陣＝右、敵陣＝左）が固定で表示される |
| **敵の出現と進行** | 1ゲームで敵が10体、左から右へ自陣方向に進む |
| **表示** | 現在の入力対象（ふりがな＋英語大文字ローマ字）は上部UIの枠内に表示する。プロトタイプでは敵の頭上には表示しない |
| **タイピング判定** | 入力した文字の色が白→グレーに変わり、全文字入力でその敵が撃破される |
| **結果画面** | クリア時にメッセージ表示、統計（時間・正解数・ミス数）、リトライボタンが動作する |
| **技術** | ブラウザで動作する（Kaplay + Vite） |

### プロトタイプのスコープ外（将来対応）

- 単語タイピング
- 難易度選択・ステージ分け
- サウンド・BGM
- スコアランキング・保存

## 5. 今後のロードマップ

### ぷち駆動開発の流れ

```
PLAN.md（本ドキュメント）
    ↓ ビジョン・ゴールの共有
SPEC.md … 詳細仕様・ロジック（画面遷移、敵の挙動、判定ルールなど）
    ↓ 実装の指針
TODO.md … タスクの分解と進捗管理（画面単位・機能単位のチェックリスト）
    ↓ 開発と並行
KNOWLEDGE.md … 学び・ハマりポイント・技術メモの記録
```

### 開発の進め方（想定）

1. **PLAN.md 確定**（本ステップ）  
   目的・ターゲット・コア体験・プロトタイプゴール・ロードマップを固定する。

2. **SPEC.md 作成**  
   スタート／ゲームプレイ／結果の各画面、敵の出現・進行、タイピング判定、結果統計などの**詳細仕様**を書き、実装の一意性を確保する。

3. **TODO.md でタスク化**  
   SPEC を「やることリスト」に分解し、優先順位と完了条件を明示する。

4. **実装と KNOWLEDGE.md**  
   実装中に気づいたこと・つまずいた点を KNOWLEDGE.md に随時メモし、今後の拡張や振り返りに活用する。

5. **プロトタイプ完了判定**  
   本 PLAN の「プロトタイプのゴール」をチェックリストとして使い、すべて満たした時点でプロトタイプ完了とする。

### テスト方針（TDD の適用範囲）

**判定・状態・ビジネスロジックだけ TDD** で進める。

- **ユニットテストで書くもの**  
  以下は **Kaplay に依存しない純粋なロジック**として切り出し、ユニットテスト（例: Vitest）でテストする。
  - **タイピング判定**: 正解／ミス、大文字小文字無視
  - **撃破条件**: 全文字入力で撃破となる条件
  - **クリア時間・正解数・ミス数の計算**: 結果画面で表示する統計の算出
- **TDD の対象外**: Kaplay の描画・シーン・オブジェクト、画面レイアウト、アニメーションなど、**見た目や描画に依存する部分**。これらは手動確認やビルド確認で担保する。
- **目的**: バグを減らしつつ、プロトタイプの作業スピードを維持する。
